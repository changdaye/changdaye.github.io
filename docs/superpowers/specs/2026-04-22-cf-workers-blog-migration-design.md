# Cloudflare Workers 静态博客迁移设计

- 日期：2026-04-22
- 目标仓库：`changdaye.github.io`
- 当前状态：Jekyll 风格博客（`_posts/`、`_layouts/`、`_includes/`、`_config.yml`）
- 目标形态：现代静态博客 + Cloudflare Workers 静态托管/路由

## 1. 背景与目标

当前仓库是一个历史较久的 Jekyll 风格博客，内容资产主要包括：

- `124` 篇 Markdown 文章，frontmatter 结构较统一
- `img/` 下的一批本地封面图、头像、背景图
- 文章中的大量外链图片，其中多数已位于阿里云 OSS：`https://cdn-blog.oss-cn-beijing.aliyuncs.com/...`
- 部分历史外链，包括 `sinaimg.cn`、`raw.githubusercontent.com`、旧 `jetbrains.org.cn` 域名内容

本次迁移目标不是继续维护旧 Jekyll 体系，而是将博客升级为更现代、长期可维护的静态博客，同时采用 Cloudflare Workers 作为部署与访问入口。

## 2. 明确约束

经确认，本次设计遵循以下约束：

1. **允许升级成更现代的博客实现**，不要求保留原有 Jekyll 主题和构建方式。
2. **博客仍然是静态站点**，不采用 Worker SSR。
3. **文章与站点图片尽量本地化为静态资源**，随站点一起构建和部署，不迁移到 Cloudflare R2。
4. **旧 URL 不要求 100% 保持不变**，以“资源和内容可访问”为主要目标。
5. **优先保证内容迁移成功与后续可维护性**，而不是对旧站做像素级复刻。

## 3. 方案对比

### 方案 A：Astro + Cloudflare Workers 静态资产托管（推荐）

使用 Astro 作为新的静态博客生成器，构建输出静态产物，再通过 Cloudflare Workers 静态资产模式托管。

优点：
- Markdown 驱动内容站点很合适
- 组件化、现代化程度高
- 后续扩展标签、RSS、Sitemap、搜索更顺手
- 仍然保持纯静态站点，稳定简单

缺点：
- 需要重建主题层
- 需要一次性迁移脚本

### 方案 B：Eleventy + Cloudflare Workers 静态资产托管

优点：
- 更接近 Jekyll 的内容站心智模型
- Markdown/frontmatter 迁移相对自然

缺点：
- 现代组件化能力与生态体验不如 Astro
- 未来继续升级样式和交互时可塑性略差

### 方案 C：保留 Jekyll，仅替换部署方式

优点：
- 改动最小

缺点：
- 无法真正摆脱旧主题、旧依赖、旧构建方式
- 不符合本次“升级成现代博客”的目标

### 结论

采用 **方案 A：Astro + Cloudflare Workers 静态资产托管**。

## 4. 总体架构

### 4.1 仓库结构策略

不在原 Jekyll 结构上直接硬改，而是在仓库中新增现代站点目录，并保留原始内容作为迁移输入源。

建议目标结构：

```text
changdaye.github.io/
├── _posts/                      # 保留旧文章，作为迁移输入源
├── img/                         # 保留旧本地图片资源
├── app/                         # 新 Astro 站点工程
├── scripts/                     # 迁移脚本与辅助脚本
├── docs/
│   └── superpowers/
│       └── specs/
└── ...
```

这样做的收益：
- 可渐进迁移，不破坏原始内容
- 可随时回看旧内容与旧模板
- 新旧体系职责清晰，降低回退成本

### 4.2 运行与部署模型

最终站点运行模型如下：

1. Astro 负责构建博客静态页面
2. 构建产物输出到 `dist/`
3. Cloudflare Workers 使用静态资产模式托管 `dist/`
4. Worker 仅承担轻量职责：
   - 静态资源分发
   - 404 兜底
   - 必要的缓存头/安全头
   - 后续可扩展少量重定向逻辑

本期不引入：
- SSR
- 数据库
- 动态后端
- 复杂 API 层

## 5. 内容迁移设计

### 5.1 内容源

将现有 `_posts/*.md` 视为迁移输入源，不直接手工逐篇重写。

已确认当前文章 frontmatter 基本统一，主要字段包括：
- `title`
- `subtitle`
- `date`
- `author`
- `header-img`
- `tags`
- `catalog`

### 5.2 迁移方式

实现一个脚本化迁移流程，负责：

1. 扫描 `_posts/*.md`
2. 解析 Jekyll frontmatter
3. 映射到 Astro 可消费的内容结构
4. 规范化 slug 与路由字段
5. 修正文章中的资源链接
6. 输出到 `app/src/content/` 或等价内容目录

迁移后的文章仍然保持 Markdown 驱动，不将文章内容硬编码到模板中。

### 5.3 路由策略

新站不强制复用 Jekyll 的全部历史 URL 规则，而采用更清晰的现代路由，例如：

```text
/posts/<slug>/
```

说明：
- 本次目标是“内容与资源可访问”，而不是“历史 URL 100% 不变”
- 如后续发现搜索流量或旧链接兼容要求更高，可补充定向重定向清单

## 6. 图片与外部资源策略

### 6.1 图片本地化总策略

本次迁移不再把阿里云 OSS 视为最终运行依赖，而是尽量将文章与站点所需图片下载到仓库内，作为静态资源随 Astro 构建结果一起发布。

目标效果：

- 头像、背景图、封面图等站点图片与文章插图一样，都作为本地静态资源存在
- Cloudflare Workers 直接分发这些静态文件
- 页面渲染尽量不依赖第三方图床在线可用性

### 6.2 阿里云 OSS 图片

对于已位于阿里云 OSS 的图片，迁移脚本优先尝试下载并本地化。

例如：

```text
https://cdn-blog.oss-cn-beijing.aliyuncs.com/17-10-12/14403343.jpg
```

迁移后的目标是将其落到站内静态目录，例如：

```text
app/public/images/posts/17-10-12/14403343.jpg
```

并把文章中的引用改写为站内路径。

### 6.3 本地图片资源

对当前仓库中的 `img/` 目录资源：
- 继续纳入新站点静态资源目录
- 用于首页背景图、封面图、头像、默认视觉素材

### 6.4 其他历史外链

当前仓库中还存在少量非 OSS 图片与历史外链，例如：
- `sinaimg.cn`
- `raw.githubusercontent.com`
- 旧 `jetbrains.org.cn` 域名相关内容

处理策略：

1. **优先下载并本地化可访问图片**
2. **自动将文章中的图片引用替换为站内静态路径**
3. **对无法下载、超时、失效的资源生成清单，单独补救**
4. **对纯网页链接、文档链接等非图片外链保持原样**

### 6.5 本地静态目录建议

建议按语义拆分静态资源目录：

```text
app/public/
├── images/
│   ├── site/        # 头像、背景图、默认视觉素材
│   └── posts/       # 文章内图片
└── favicon/...
```

### 6.6 内容中的旧站内链

对于文章正文里的旧域名站内链接：
- 若能判断其对应当前博客文章，则自动转换为新站内链
- 若其本质是历史服务地址、教程示例或外部服务入口，则按原文保留

## 7. 页面结构与功能范围

### 7.1 首期页面

首期版本包含：

1. **首页**
   - 博客标题、简介
   - 最新文章列表
   - 文章摘要卡片

2. **文章详情页**
   - 标题、副标题、日期、作者
   - Markdown 正文渲染
   - 封面图显示
   - 目录（基于标题层级）
   - 上一篇 / 下一篇

3. **标签页**
   - 所有标签列表
   - 标签下文章列表

4. **About 页面**
   - 迁移现有 `about.html` 的内容

5. **404 页面**
   - 友好兜底

6. **RSS / Sitemap**
   - 支持订阅与搜索引擎收录

### 7.2 首期不做

为降低迁移风险，首期不引入以下旧功能：

- Gitalk 评论
- 百度分享
- 多说 / 畅言等旧评论系统
- 百度统计等旧时代分析脚本
- 旧第三方挂件

原因：
- 很多依赖已过时或侵入性强
- 会增加兼容性和维护负担
- 与“先把内容博客稳定迁走”的目标冲突

## 8. 视觉与信息架构

### 8.1 视觉方向

新站风格应以“现代技术博客”为主，优先保证：
- 排版可读性
- 移动端体验
- 轻量性能
- 简洁层次

不要求复刻旧主题，也不追求复杂动画。

### 8.2 导航结构

首期导航建议保持极简：
- 首页
- 标签
- About
- GitHub

这样既满足当前博客需求，也为后续增加归档、项目页、搜索保留空间。

## 9. Cloudflare 集成设计

### 9.1 部署方式

采用 Wrangler 驱动部署，结合 Cloudflare Workers 静态资产能力。

目标结果：
- 本地可开发、可预览
- 可通过 `wrangler` 构建/部署
- 将来可直接绑定自定义域名

### 9.2 Worker 职责边界

Worker 只保留轻量边界：
- 静态产物托管
- 404 处理
- 基础缓存/响应头
- 未来少量重定向支持

不在本项目中引入业务型 API 逻辑。

## 10. 实施步骤

建议按以下顺序实施：

1. 新建 `app/` Astro 工程
2. 接入 Cloudflare Workers 部署配置
3. 先实现首页 / 文章页 / 标签页 / About / 404 基础骨架
4. 编写 `_posts` → 新内容格式的迁移逻辑
5. 迁移 `img/` 本地资源引用
6. 下载并本地化 OSS 与其他可访问文章图片，清洗图片引用
7. 接入 RSS / Sitemap
8. 完成本地验证与部署文档

## 11. 验证策略

迁移实施后，至少完成以下验证：

1. 本地依赖安装成功
2. 本地开发预览成功
3. 构建成功
4. 抽样验证多篇文章：
   - 中文标题正常
   - frontmatter 映射正常
   - tags 正常
   - header image 正常
   - 本地化后的文章图片正常显示
5. About 页面正常
6. 标签页正常
7. RSS / Sitemap 可生成
8. Wrangler 本地运行成功
9. Cloudflare 部署配置可直接使用
10. 下载失败图片清单可输出并复核

## 12. 风险与后续事项

### 12.1 已知风险

1. 部分历史外链可能已失效，无法自动下载
2. 少量旧域名站内链可能无法自动 100% 准确映射
3. `about.html` 与旧模板中的少量脚本内容需要筛除历史无效依赖
4. 个别 Markdown 文章可能包含旧 Jekyll/HTML 写法，需要兼容处理
5. 本地化全部文章图片会增加仓库体积，需要接受仓库变大

### 12.2 后续可扩展项

本期完成后，可按需补充：
- 搜索
- 归档页
- 更细粒度的重定向规则
- 评论系统（如确有需要，重新选择现代方案）
- 自定义域名和缓存细化配置

## 13. 结论

本次迁移将把 `changdaye.github.io` 从历史 Jekyll 博客升级为：

- **现代静态博客工程**
- **基于 Astro 构建**
- **通过 Cloudflare Workers 托管**
- **将文章与站点图片本地化为静态资源**

设计重点是：
- 保住文章内容
- 尽量摆脱外部图床依赖
- 摆脱旧主题与旧构建方式
- 为后续长期维护和继续演进打基础
